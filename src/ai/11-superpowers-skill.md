# Superpowers：我为什么开始让 AI 按工程流程干活

> 对 iOS / Flutter 开发者来说，AI Coding 最麻烦的地方往往不是“它不会写”，而是“它太快就开始写”。我开始用 Superpowers，就是想让它先慢半拍。

---

## 前言

我现在越来越少把 Coding Agent 当成一个“更聪明的搜索框”。

在移动端项目里，这种用法不太够。一个需求可能同时碰到 Flutter 页面、iOS 原生桥接、CocoaPods、Gradle、路由、埋点、权限、网络层和老代码兼容。你让 AI 直接改，它大概率能给出一版看起来能跑的代码。麻烦在后面。

它可能还没看项目里的已有写法，就先造了一套新结构；可能还没搞清楚 bug 的来源，就先包了一层异常处理；也可能改完就告诉你“已经修复”，但没有任何验证证据。

用了一段时间 Superpowers 之后，我对 AI Coding 的看法有点变了。模型能力当然重要，但工作流不能缺。

---

## 一、Superpowers 是什么

Superpowers 不是某个单一能力，也不是一句万能提示词。

它更像是一套给 Coding Agent 用的工程习惯，拆成了一组 skills：需求澄清、方案设计、写计划、TDD、系统化调试、代码审查、完成前验证、分支收尾。

它的思路很直接：不要让 AI 一上来就写代码，而是让它按工程习惯走。

比如：

```
有新需求 → 先澄清和设计
要改代码 → 先写计划
遇到 bug → 先找根因
要实现功能 → 先写失败测试
准备收尾 → 先跑验证
```

这些事情本来就不新。难得的是，它把这些习惯写进了 Agent 的工作方式里。

---

## 二、移动端项目为什么特别需要这种约束

纯前端页面、小脚本、小工具，AI 写错了通常比较好发现。

移动端项目不一样。尤其是 iOS + Flutter 混合项目，很多问题不是一眼能看出来的。

一个 Flutter 页面加载失败，原因可能在 Dart 侧状态管理，也可能在 iOS 侧 MethodChannel。再往下查，还有路由参数、权限申请、原生 SDK 初始化顺序，甚至某个环境变量没有传进构建脚本。

这种时候，AI 最容易犯的错是“看到症状就补一刀”。

页面空白？加个 loading。

接口失败？包一层 try-catch。

原生回调没触发？换一种写法再试试。

这些改法有时能蒙对，但更多时候只是把问题藏得更深。Superpowers 里的 systematic-debugging 会要求它先做根因调查：复现、看错误、查最近改动、追数据流、找同类代码。这个流程不酷，但像一个靠谱工程师会做的事。

这对移动端开发很实用。移动端 bug 经常跨层，靠猜很容易越修越乱。

---

## 三、我最喜欢的是：它会先停一下

很多 AI 工具的问题，是太积极了。

你说“帮我修一下”，它马上开始改。

你说“这个页面加个入口”，它马上补路由、补按钮、补页面。

你说“这里报错”，它马上给出一个修复方案。

但真实开发里，很多时候第一步不是写代码，而是确认一些很细的问题：这个需求到底要改哪里？项目里有没有现成模式？这是不是平台差异？有没有历史原因？改完怎么证明没影响别的地方？

Superpowers 的 brainstorming 和 writing-plans 就卡在这里。

它会把一个模糊需求先拆成设计，再拆成可执行步骤。对我这种经常在 iOS、Flutter、服务端接口文档之间来回切的人来说，这一步能少掉很多“AI 自己脑补”的空间。

尤其是老项目。老项目最怕的不是代码慢，而是 AI 很自信地引入一套新风格。

移动端项目里这种情况很常见。项目本来用 Provider，它突然写 Riverpod；项目路由有统一封装，它直接 `Navigator.push`；iOS 里已有封装好的 bridge，它重新写一套 MethodChannel。

流程不是为了拖慢 AI，是为了让它先看路。

---

## 四、TDD 这件事，在移动端也有价值

Superpowers 里最“硬”的部分之一，是 test-driven-development。

它的要求很严格：先写测试，确认测试失败，再写最小实现，最后确认测试通过。

说实话，移动端开发者第一次看到这种要求，可能会有点抗拒。Flutter Widget Test、iOS 单测、集成测试都不是永远顺手。很多项目历史包袱也不小，补测试并不轻松。

但我现在更能理解它为什么这么坚持。

AI 写代码的速度太快了。没有测试时，这种速度反而会变成风险。你很难靠肉眼确认所有边界都对，更别说它还可能改到你没注意的地方。

在移动端里，我不会机械地要求每个 UI 像素都先写测试。但下面这些场景，我会更愿意让 AI 走 TDD：

- 数据解析
- ViewModel / Controller 逻辑
- Repository 分支
- 权限状态流转
- 路由参数处理
- 原生桥接的入参出参转换

这些地方一旦写错，后面排查很麻烦。让 AI 先写一个失败测试，至少能把“它到底在实现什么行为”钉住。

---

## 五、我现在会逼它说清楚：验证了什么

我现在对 AI 最警惕的一句话是：“已经修复。”

人类工程师说这句话，通常也要带证据：跑了哪个测试、看了哪个日志、复现步骤现在是什么结果、有没有影响相关页面。

AI 更需要这个约束。

Superpowers 的 verification-before-completion 把这件事说得很硬：没有刚跑出来的验证结果，就不能说完成。

这点很适合移动端。因为很多移动端问题不是编译过了就完事：

- Flutter analyze 过了，不代表页面交互没问题
- iOS build 过了，不代表真机权限流程没问题
- 单测过了，不代表某个渠道包配置没问题
- 接口 mock 过了，不代表线上数据结构没坑

所以我更喜欢让 Agent 在结束前说清楚：跑了什么，结果是什么，哪些没跑，风险在哪里。

这比“多写一点代码”更有用。

---

## 六、我理解为什么很多人会用它

我觉得 Superpowers 会被不少开发者接受，不是因为它发明了什么新概念。

它解决的，其实是很多人用 AI 写代码时的同一个痛点：AI 太会动手，但不太会收手。

Superpowers 把很多老派工程习惯重新塞回来了：

- 做需求前先问清楚
- 写代码前先看现有模式
- 调 bug 先找根因
- 实现功能先考虑测试
- 说完成前先验证
- 大任务拆小步
- 让不同子任务隔离执行

这些习惯在团队里本来就有。人类工程师会靠经验记住，AI 不会。AI 需要这些流程被写下来、被触发，甚至被强制执行。

这也是 Skills 这种机制有意思的地方。它不是给 AI 多接一个外部系统，而是把团队工作方式写进上下文。

大家已经不满足于“AI 能写代码”了。更现实的问题是，它能不能像一个靠谱工程成员那样工作。

---

## 七、我会怎么用它

我不会把 Superpowers 当成所有任务的固定模板。

小改文案、查一个简单 API、改一行样式，完整流程可能太重。工具再好，也不能盖过任务本身。

但下面这些场景，我会倾向于让 Agent 把流程走完整：

- 新功能跨多个模块
- iOS 和 Flutter 都要改
- 老项目里不确定历史逻辑
- bug 复现路径复杂
- 修改会影响支付、登录、权限、路由、数据同步
- 需要 AI 连续工作一段时间，而不是只回答一个问题

我的判断很简单：任务越容易失控，Superpowers 越有价值。

当然，它不是免费午餐。流程越完整，前期沟通、写计划、补测试、跑验证的成本也越高。对一个没什么测试基础的老移动端项目来说，刚开始用它可能会有点别扭。

---

## 写在最后

我以前用 AI 写代码，更关注“它会不会”。

现在我更关注“它怎么做”。

会写代码只是开始。影响交付质量的，是它有没有先理解问题，有没有尊重项目结构，有没有找根因，有没有测试，有没有验证。

Superpowers 解决的就是这个问题。它不是让 AI 突然变聪明，而是让 AI 少一点冲动，多一点工程纪律。

这对 iOS / Flutter 开发者很现实。

因为我们面对的不是干净的新项目，而是混合架构、历史包袱、平台差异，还有很多不能随便坏的业务流程。AI 可以帮我们写得更快，但前提是它不能乱跑。

只看输出速度，AI 已经够快了。接下来我更关心的是，它能不能稳一点。
