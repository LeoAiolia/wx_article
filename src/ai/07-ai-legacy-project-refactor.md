# AI Coding 移动端工程实践（五）：让 AI 改老项目，如何避免越改越乱？

> 老项目最怕的不是 AI 不会写，而是它太会写，顺手把历史问题扩散到更多地方。

---

## 前言

老项目通常有几个特点：

- 目录结构复杂
- 历史写法不统一
- 相似页面很多
- 有些代码不能模仿
- 一些逻辑只能先兼容，不能马上重构

这种项目里用 AI，第一原则不是快，而是稳。

---

## 一、先读，不急着改

让 AI 改老项目，第一步应该是：

```
请先阅读相关文件，说明当前实现方式和风险。
不要修改代码。
```

先让它理解现状，再让它提出方案。

如果一上来就改，很容易把局部问题变成全局问题。

---

## 二、标出不要模仿的旧代码

老项目里常有一些历史写法：

- 页面直接调 API
- UI 和业务逻辑混在一起
- MethodChannel 里塞业务判断
- Flutter Widget 巨大且不可复用
- iOS 原生和 Flutter 路由互相绕

这些要在项目索引里标出来：

```
以下代码是历史遗留，只能兼容，不建议新代码模仿。
```

AI 很容易学习现有代码。如果现有代码有坏味道，就要明确告诉它不要学。

---

## 三、限制修改范围

给 AI 的任务要写清楚：

```
本次只修复订单详情页的状态刷新问题。
不要重构网络层。
不要修改路由表。
不要调整 UI 样式。
不要新增公共组件。
```

越是老项目，越要限制范围。

---

## 四、先做适配，再谈重构

很多老项目不能直接推倒重来。

更现实的策略是：

- 先修 bug
- 再补测试
- 再提取重复逻辑
- 最后逐步替换旧实现

不要让 AI 一次性「顺手优化架构」。这通常会带来更大的风险。

---

## 五、给老项目建立保护网

改老项目之前，最好先补几类保护：

- 关键路径手动 checklist
- ViewModel / Controller 状态测试
- Repository 错误分支测试
- Deep Link 回归路径
- Flutter / 原生跳转验证

有了保护网，再让 AI 改代码会安全很多。

---

## 写在最后

老项目不是不能用 AI，而是更需要规则。

让 AI 先读、先解释、先出方案，再小步修改、严格验证。这样它才能帮你减轻负担，而不是制造新的历史包袱。
