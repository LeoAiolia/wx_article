# 从苹果供应链事件，聊聊软件团队的安全边界

> 把钥匙交出去的那一刻，门还安不安全，就不全是你说了算了。

---

最近发生了两件事。

第一件。2026 年 6 月，苹果在印度的供应链伙伴 Tata Electronics 发生数据安全事件，部分疑涉苹果产品制造、测试和供应商信息的资料被发布到网络。Tata 后来确认了网络安全事件的存在，但未公开确认所有流传资料的范围；Reuters 则说明，它无法独立验证全部数据的真实性。

第二件。2026 年 7 月 3 日，科创板日报和第一财经报道了阿里内部禁用 Claude Code 的消息。据内部人士透露，阿里将 Claude Code 列入高风险软件名单，自 7 月 10 日起禁止员工在办公环境使用，并推荐 Qoder 作为替代。

一条是硬件供应链泄露，一条是软件工具被禁用。看起来不搭边。但放在一起，指向同一个问题：你信任的那些人和工具，还安全吗？

---

世人谈安全，眼睛总盯着正门。

服务器有没有被攻破，防火墙有没有漏洞。这些当然重要。但 Tata 这件事提醒了另一面：有时候，问题不出在自家的城墙，出在隔壁那扇你给了钥匙的偏门。出事的不是苹果，是它信任的人。

这让我想起一个很老的安全原则。链条的强度，取决于最薄弱的那一环。那一环经常不是你自己。

一部 iPhone 背后是盘根错节的协作关系：谁供哪个零部件，谁是备选，哪些环节全世界只有两三家公司能做，哪些测试在哪些工厂跑。外观迟早发布，参数迟早公布。但"某个未发布产品的某个部件交给谁做"，这种信息永远不会出现在苹果官网的供应商名单里。一个是透明，一个是商业机密。两回事。

硬件如此，软件也没好到哪去。

软件行业里，供应链换了一堆名字：第三方 SDK、CI/CD 平台、崩溃日志、测试分发、外包团队、设计协作工具。现在还多了一个角色，AI Coding Agent。它读你的代码，理解你的业务，调用你的命令，连远程模型。它早就不是编辑器插件了。它在代码、密钥和构建环境的交叉点上干活。阿里把 Claude Code 列入高风险名单，担心的无非也是同一件事：工具能看到什么、能传走什么、能执行什么。

回看自己身边的事情，有些不复杂，但总会被忽略：

离职同事的仓库权限注销了吗。API token 上次轮换是什么时候。测试包能不能被无限转发。SDK 到底采集了什么、传到了哪里。AI 工具能不能读敏感文件，能不能执行命令。

这些问题不酷，不属于任何需求。但历史反复讲同一件事：出事的，往往是那些大家默认没问题的角落。

安全不是一道围墙。是一张信任网络。网络里每个节点，供应商、外包方、工具平台、AI Agent，都在帮你做事，也都在替你开门。你修得好自家的服务器，修不了链条上每个人的习惯。

下次安全会议，别只问"我们有没有被攻击"。

多问一句：交出去的那些权限，现在还安全吗？
